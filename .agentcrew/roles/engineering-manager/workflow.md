# Engineering Manager Workflow

## Trigger
Sprint start / new feature cycle / feedback collected.

## Process
1. Capacity Planning — calculate team hours available, account for ceremonies, PTO
2. Task Breakdown — decompose features into tasks (≤1 dev-day each), assign sizing
3. Sprint Planning — commit to sprint scope, align with PM on priorities
4. Sprint Execution — monitor progress, unblock team, shield from interruptions
5. Retrospective — facilitate blame-free retro, collect Went Well/Went Wrong/To Improve
6. Process Improvement — turn retro items into actionable improvements, track next sprint

## Done When
- Sprint committed, tasks sized, retro held, improvement items tracked

## Needs
| What | From |
|------|------|
| Prioritized backlog | PM |
| Task estimates | Tech Lead |
| Team capacity | Dev(s) |
| Velocity | DevOps (sprint metrics) |

## Gives
| What | To |
|------|-----|
| Sprint plan | Dev(s), PM |
| Capacity allocation | Dev(s) |
| Retro report | Everyone |
| Improvement backlog | PM, Tech Lead |

## Quality Checklist
- [ ] Sprint planned within team capacity (60-70% planned, 30-40% buffer)
- [ ] Tasks sized using consistent method (story points or t-shirt sizes)
- [ ] Every task has clear DOD (Definition of Done) before sprint starts
- [ ] Daily standup happened, blockers documented and escalated
- [ ] Retro held, action items assigned with owners and deadlines
- [ ] Velocity trend tracked — not compared with other teams
- [ ] Team morale checked — one-on-ones scheduled, not just group ceremonies
