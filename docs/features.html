Zero-Config Entry New

A single entry point for any user — describe what you need in plain English, no CrewMarkdown knowledge required.

Who it's for: Entrepreneurs, founders, product managers, or anyone who wants results without learning about objectives, squads, or role contracts.

Usage

./go.ps1
> Build me a landing page for my SaaS product

Behind the scenes, the full CrewMarkdown pipeline runs: semantic routing, squad assembly, contract loading, workflow execution, quality gates — all invisible to the user.

Key Design

  • Jargon-free: User never sees "objective", "squad", or "contract"
  • Full power under the hood: Same pipeline powers both zero-config and expert modes
  • Progressive disclosure: Users can start with zero-config and gradually learn role commands
  • Auto-creates config: First run generates .crewmarkdown/state/zero-config.json

Skill-Level Inference New

CrewMarkdown reads your input and adjusts agent autonomy automatically — same system, different experience per user.

No configuration required. The agent infers skill level from 5 input signals and adapts mid-session based on your follow-ups.

3 Modes

ModeAgent AutonomyUser InvolvementTypical Input
Auto Full — decides objectives, spawns roles, produces all artifacts Passive — approves/rejects outcomes Vague, high-level, no tech jargon
Guided Medium — proposes plan, explains options, seeks input Active — chooses between options, provides feedback Goals with plain language or questions
Expert Low — assumes domain knowledge, executes precisely Driving — gives detailed instructions Role names, tech jargon, explicit commands

Inference Signals

The agent computes a specificity score (0-100) from 5 weighted signals:

SignalWeightAuto SignalExpert Signal
Role mention30%None ("build my app")Explicit ("Architect, design")
Jargon density25%0 tech terms3+ specific terms
Verb specificity20%"I want", "make""implement", "configure"
Scope granularity15%Whole projectSpecific task
Question format10%Open-endedImperative

Dynamic Switching

Mode is not static — it adapts mid-session. If an auto-mode user asks a specific question, the agent flexes toward expert for that exchange. If an expert user says "explain like I'm 5", it flexes toward guided. Recent input is weighted 2x for responsiveness.

Sub-Specialization

58 sub-specializations across 17 role families enable per-request role refinement via hybrid keyword resolution — the right contract for the right context, every time.

Rationale: A "Frontend Developer" working with React requires a fundamentally different system prompt than one working with Svelte. Sub-specs load the appropriate contract per request without bloating or duplicating the parent role definition.

Resolution Algorithm

The routing engine implements a hybrid resolution strategy: scan request for keywords: in each sub-spec's YAML frontmatter, apply layer × framework matching, and fall back to the generic parent contract when no sub-spec match is found — ensuring zero flow interruption.

  1. Keyword scanning — Agent scans request text against keywords: fields in each sub-spec's YAML frontmatter
  2. Layer × Framework matching — Match on both architectural layer and specific framework/technology for precise resolution
  3. Sub-spec loading — Exact match loads sub-spec contract.md + workflow.md; parent role cached as fallback
  4. Graceful fallback — No match resolves to generic parent contract (never breaks execution flow)

Full algorithm documented in .crewmarkdown/procedures/role-subspec.md

Debate Framework

Structured decision-making framework for resolving disagreements when multiple viable options exist. Ensures all perspectives are evaluated on merit with documented rationale.

Cross-cutting: Can be activated at decision points within any lifecycle objective.

Purpose

When multiple valid approaches exist for a decision, the debate framework ensures all perspectives are systematically evaluated, trade-offs are explicitly documented, and the optimal option is selected based on agreed scoring criteria rather than bias or recency.

Process Flow

1. Debate Trigger → 2. Panel Assembly → 3. Opening Statements → 4. Cross-Examination → 5. Scoring → 6. Resolution
  1. Debate Trigger — A decision point is identified with multiple viable options. The objective lead calls for a structured debate.
  2. Panel Assembly — Facilitator assembles roles with a stake in the decision. Each panelist represents their domain perspective.
  3. Opening Statements — Each panelist presents their position with supporting evidence and reasoning.
  4. Cross-Examination — Structured rounds of rebuttal, questioning, and evidence presentation.
  5. Scoring — Panelists evaluate options against agreed criteria using a weighted scoring system. Highest aggregate score wins.
  6. Resolution — Decision is formally recorded with rationale, rejected alternatives, and expected consequences.

Production incident management from first signal through resolution and blameless postmortem. Activated via /incident [signal].

Cross-cutting: Can be activated at any point when a production issue is detected.

Structured Review Mode

Synchronous or asynchronous review of any artifact type with findings classification, discussion, and formal sign-off. Activated via /review [artifact].

Cross-cutting: Applicable to code, architecture, design, PRDs, test plans, documentation, and security artifacts.

Process Flow

1. Prepare → 2. Present → 3. Discuss → 4. Decide → 5. Log

Findings classified as Blocking (must fix), Major (should fix), Suggestion (nice to have), or Question (clarification). Outcomes: Approved, Changes Required, or Rejected.

Structured Interview Mode

Plan, conduct, and synthesize structured interviews with users, stakeholders, or subject matter experts. Activated via /interview [topic].

Cross-cutting: Supports user research, stakeholder discovery, hiring interviews, and expert consultations.

Process Flow

1. Plan → 2. Conduct → 3. Synthesize → 4. Decide → 5. Log

Supports structured, semi-structured, and unstructured formats. Produces interview summaries with key insights, themes, notable quotes, and actionable recommendations.

Facilitated Negotiation Mode

Structured negotiation between parties with conflicting interests to reach mutually acceptable agreements. Activated via /negotiate [issue].

Cross-cutting: Useful for scope vs timeline trade-offs, budget allocation, resource conflicts, and technical approach disagreements.

Process Flow

1. Prepare → 2. Discuss → 3. Negotiate → 4. Agree → 5. Log

Uses principled negotiation techniques: interest-based bargaining, BATNA analysis, objective criteria, and trade-off packaging. Produces documented agreement terms with sign-off.

Retrospective Mode

Blameless team retrospectives for sprints, releases, or project phases. Activated via /retro [period].

Cross-cutting: Can run at any cadence — sprint, release, quarterly, or ad-hoc after incidents.

Process Flow

1. Set Stage → 2. Gather Data → 3. Generate Insights → 4. Decide Actions → 5. Log

Supports multiple formats: Start/Stop/Continue, Glad/Sad/Mad, 4 L's, Sailboat, Timeline. Root cause analysis via 5 Whys and fishbone diagrams. Produces SMART action items and improvement experiments.

Hackathon Mode

Time-boxed innovation sprints with pitch, team formation, build, demo, and judging. Activated via /hackathon [theme].

Cross-cutting: Ideal for innovation weeks, internal tool development, proof-of-concept sprints, and team building.

Process Flow

1. Pitch & Form Teams → 2. Plan & Setup → 3. Build → 4. Demo & Judge → 5. Follow-up

Judging criteria: Innovation (25%), Impact (25%), Execution (25%), Presentation (15%), Teamwork (10%). Special categories for Best Technical Achievement, Most Creative, Best UX, People's Choice.

Tactical Mode

Continuous always-on mode that stays active for the entire thread, dynamically dispatching roles per request with persistent context. Activated via /tactical.

Cross-cutting: Once activated, remains live across the whole conversation — routes every request to the right role(s), maintains thread state, logs decisions, and produces tactical summaries.

Process Flow

1. Activate → 2. Dispatch → 3. Track → 4. Decide → 5. Summarize

The Tactical Commander initializes thread state on activation, dispatches roles dynamically for each request, maintains an action register and decision log, and produces tactical summaries on demand or thread end.

Onboarding Mode

Structured new hire ramp-up from day 1 through independent contribution. Activated via /onboard [role].

Cross-cutting: Works for any role — engineer, designer, PM, marketer, or executive.

Process Flow

1. Plan → 2. Orient → 3. Learn → 4. Check → 5. Ramp

Weekly milestones: Week 1 — setup + first task, Week 2 — first PR merged, Week 3 — picks up tasks independently, Month 1 — owns small feature end-to-end, Month 2 — unblocks self. Produces onboarding completion report with feedback for team improvement.
